# 素体差リファレンス(アバター製品ごとの設計差と調査手順)

ツール固有ではなく「どの素体(アバター製品)を触るか」で変わる知識を扱う。素体ごとの設計思想の違いに起因する罠は、ツールの使い方が正しくても踏む。ここには**類型(クラス)・実例の最小限・改変前の判定手順**を置き、個別製品の網羅仕様は置かない。

> 情報源: 実測(Unity 2022.3.22f1 / VRCSDK3、2026年前半)。実例として挙げる製品名は類型の根拠であり、製品評価ではない。

## ブレンドシェイプ命名の設計分類

素体を跨ぐ衣装流用や BlendShape 同期の成立性は**素体の設計思想に依存**し、一律「互換/非互換」ではない。実測に基づく類型:

### 同名前提の設計(実例: ミルティナ / DOLOSart製、2026-07-10実測)

- 衣装の各メッシュ(エプロン・下着・ドレス等)に**体側と同名の体型BS**(例: `Breasts_Flat/Small/Big/Cow`)を体系的に配置
- オプション変形はプレフィックス(例: `Option_*`)で統一
- 「同名BSが同期元/先にある」前提の手順(MA Blendshape Sync のデフォルト運用等)がそのまま成立する
- **注意(衣装プレフィックス型の派生)**: 英語体系でも、衣装メッシュ側の体型BSが素体と同名(`Breasts_big`)ではなく**衣装名プレフィックス付き**(`<衣装>_breasts_big` 等)で重複配置される設計がある(実例: ライム、2026-07-11実測)。この場合 MA Blendshape Sync のデフォルト同名同期は素体↔衣装を橋渡ししない。体型は製品側アニメで一括駆動される想定で、素体を跨いで同期したいときのみ `LocalBlendshape` で対応先を明示する。

### 独自命名の設計(実例: ルルネ / IKUSIA製、2026-07-10実測)

- 日英混在・記号混じり(例: `Breast_small_____胸_小` のようにアンダースコア連発+日本語サフィックス)で、素体間の同名前提が成立しない(まめひなた等の英語名体系とは完全に別)
- リターゲットツール痕跡のプレフィックスBS(例: `Retarget_Body_b_...`)が残存することがある
- メッシュごとにBS構成が不統一なことがある

### 改変前の判定手順

1. 改変対象の衣装・アバター各 SkinnedMeshRenderer の BS 名一覧をダンプする(スクリプトなら `mesh.GetBlendShapeName(index)` 全件ダンプ → テキスト検索が確実)
2. (a) 同名が大多数 → 同名前提の同期をそのまま張る
   (b) 独自命名・混在 → 両者の対応表を作り、同期先BS名を明示指定する(MA Blendshape Sync なら `LocalBlendshape` フィールド。[02](02-modular-avatar.md))

補足: 同一製品の衣装変種(体型違いprefab等)はBS構成・BS数が完全同一で差がベースメッシュ形状のみのことがあり、**BS構成では変種を判別できない**。プレハブ名・GameObject名で判別する(実測: 2026-07-10)。

## セパレータ(区切り)専用ブレンドシェイプ

**変形を持たない「区切り行」用BSが存在する素体がある(実測)**: 例: ルルネ(IKUSIA製、実測 2026-07-10、Unity 2022.3.22f1)の Body メッシュには `─────VRC─────`、Body_b メッシュには `==Breast_____胸======================` のような区切り専用BSが存在。これらは **weightキー・フレームなし**で変形は起きない。BSをインデックス直指定・前方一致・全件走査で一括操作すると、これらを巻き込み戻し忘れの原因になる。

**独自命名の素体に限らない**: 英語体系で系統的な素体(実例: ライム、2026-07-11実測。Body_Base の `-------Breasts-------`、Knee-Socks の `-------Separate-------`)でもセパレータ専用BSは出る。命名設計とセパレータBSの有無は独立と考えてよい。

**対策**: BSは**名前の完全一致**で引く。改変前に各 SkinnedMeshRenderer の一覧をダンプし、命名規則(区切り行の有無・日英混在)を確認する。変形キーフレーム有無での事前フィルタも有効。

## 衣装トグルのOFFリスト漏れ — 部位シグネチャによる機械判定

衣装切替トグル(MA Object Toggle)のOFFリストをオブジェクト名のみで組むと、同スロットの既定衣装(下着・靴など)を取りこぼしやすい。SkinnedMeshRendererのスキンウェイトを使った機械判定手順が有効(実測: Unity 2022.3.22f1 / MA 1.17.1 / NDMF 1.14.0)。

**手順:**

1. **部位シグネチャの生成**: 各SkinnedMeshRendererの全頂点ボーンウェイトをヒューマノイド8カテゴリ(Head系 / TorsoUpper=Chest・Spine・Shoulder / Arm / Hand / Hips / LegUpper / LegLower / Foot系)に集計・正規化し「部位シグネチャ」とする。非ヒューマノイドボーン(髪・衣装ボーン)はTransform親を遡り最初のヒューマノイドボーンのカテゴリへ合算する。
   - **衣装が独自アーマチャーを持つ場合の注意**: そのボーンは `Animator.GetBoneTransform()` が返すTransformと別インスタンスになりヒットしない。ヒューマノイドボーン名→カテゴリの名前マップをフォールバックとして用意する。
2. **スロット推定**: Foot ≥ 0.5 → 足元 / Head ≥ 0.5 → 頭 / TorsoUpper+Arm ≥ 0.6 → 上衣 / Hips+LegUpper+LegLower ≥ 0.6 → 下衣。
3. **複数メッシュの合成**: 新衣装が複数メッシュを持つ場合は**カテゴリ別MAXで合成**する。平均では下半身担当メッシュの成分が希薄化し、ショーツ・スパッツ等の競合が閾値を割る(実測で0.24〜0.27に低下)。
4. **競合判定**: 新衣装の合成シグネチャと既定衣装各メッシュのシグネチャの要素ごとのmin和(オーバーラップ)が **≥ 0.3 → 自動OFF候補**。ただし**足元スロット(Foot ≥ 0.5)は自動OFFにせず「要判断」に分類**する(新衣装が足をカバーしない場合、靴を消すかどうかは改変の意図に依存するため)。
5. **出力分類**: 「自動OFF候補 / 残す / 要判断」の3リストにまとめ、要判断のみ目視確認(スクリーンショット等)に回す。

## よくあるトラブル

- **ブレンドシェイプを一括操作したが何か壊れた/思わぬメッシュが変形した**: インデックス直指定・前方一致・全件走査で「区切り行」専用BSを巻き込んだ可能性。命名規則を確認し完全一致で引く、または変形キーフレーム有無で事前フィルタする(§セパレータ専用ブレンドシェイプ)
- **衣装を別素体のアバターに着せたが表情・体型が反応しない(BS同期無反応)**: 素体間でBS命名が互換でない可能性。両者のBS一覧を突き合わせ、名前が違う場合は同期先BS名を明示指定する(§改変前の判定手順、MAの設定方法は[02](02-modular-avatar.md))
